# PostgreSQL Database Structure — Release 1.0
## AI English Learning App

---

## Architecture Decisions

| Decision | Choice | Why |
|----------|--------|-----|
| Database | PostgreSQL 16 (Docker) | Production-ready, JSON support, enum types |
| ORM | SQLAlchemy 2.0 async | Typed models, migration-ready |
| Migrations | Alembic | Version-controlled schema, autogenerate from models |
| Driver | asyncpg | Fastest async PostgreSQL driver for Python |
| UUID Strategy | App-generated (uuid4 string) | No DB extension needed, portable |
| Timezone | All timestamps in UTC | Stored as `TIMESTAMP WITH TIME ZONE`, displayed in JST by Flutter |

## Table Schemas

### users (Week 1 ★)

The core table. Handles auth, free session tracking, and user profile.

```sql
CREATE TYPE age_group_enum AS ENUM (
    'under16', 'high_school', 'college',
    '26_35', '36_50', '51_plus'
);

CREATE TYPE english_level_enum AS ENUM (
    'beginner', 'elementary',
    'intermediate', 'upper_intermediate'
);

CREATE TYPE user_status_enum AS ENUM ('active', 'deleted');

CREATE TABLE users (
    user_id         VARCHAR(36)         PRIMARY KEY,  -- uuid4 generated by app
    email           VARCHAR(255)        NOT NULL,
    phone           VARCHAR(15)         NOT NULL,     -- +81 format: +819012345678
    password_hash   VARCHAR(255)        NOT NULL,
    display_name    VARCHAR(20),
    age_group       age_group_enum      NOT NULL,
    max_free_sessions   INTEGER         NOT NULL,     -- 5 (adult) or 10 (student)
    used_free_sessions  INTEGER         NOT NULL DEFAULT 0,
    api_key_registered  BOOLEAN         NOT NULL DEFAULT FALSE,
    english_level   english_level_enum,               -- set after onboarding
    primary_goal    VARCHAR(500),                      -- set after onboarding
    onboarding_completed BOOLEAN        NOT NULL DEFAULT FALSE,
    device_fingerprint  VARCHAR(255),
    referred_by     VARCHAR(36),                       -- prepared for Release 1.2
    created_at      TIMESTAMPTZ         NOT NULL DEFAULT NOW(),
    last_active_at  TIMESTAMPTZ         NOT NULL DEFAULT NOW(),
    status          user_status_enum    NOT NULL DEFAULT 'active',
    deleted_at      TIMESTAMPTZ,

    CONSTRAINT uq_users_email UNIQUE (email),
    CONSTRAINT uq_users_phone UNIQUE (phone),
    CONSTRAINT chk_used_sessions CHECK (used_free_sessions >= 0),
    CONSTRAINT chk_used_le_max CHECK (used_free_sessions <= max_free_sessions)
);
```

**Indexes:**
```sql
-- Unique constraints auto-create indexes on email and phone.
-- Add these only if query patterns justify them:
CREATE INDEX idx_users_status ON users (status);
CREATE INDEX idx_users_created_at ON users (created_at);
```

**Column notes:**

| Column | Format / Rules | Example |
|--------|---------------|---------|
| `user_id` | UUID v4 as string, generated by Python `uuid.uuid4()` | `a1b2c3d4-e5f6-...` |
| `email` | Standard email, unique across all accounts | `user@example.com` |
| `phone` | Always stored as `+81` format, unique | `+819012345678` |
| `password_hash` | bcrypt hash from passlib | `$2b$12$...` |
| `display_name` | Optional nickname, max 20 chars | `タロウ` |
| `max_free_sessions` | Set at registration: 5 for adults (26+), 10 for students (under 25) | `5` or `10` |
| `used_free_sessions` | Incremented only on Review phase completion (Screen C4) | `0` → `1` → `2` ... |
| `api_key_registered` | Set to TRUE when user confirms own Gemini key works | `false` |
| `english_level` | Set after onboarding Turn 3 level assessment | `beginner` |
| `primary_goal` | Set after onboarding Turn 1 goal detection | `海外旅行` |
| `referred_by` | Another user's user_id (prepared for Release 1.2) | `NULL` |
| `status` | Soft delete: set to `deleted`, don't remove row | `active` |

### deleted_accounts (Week 4)

Prevents users from deleting their account and re-registering for free sessions.
Stores only hashes — no personal data retained after deletion.

```sql
CREATE TABLE deleted_accounts (
    id                  SERIAL          PRIMARY KEY,
    email_hash          VARCHAR(64)     NOT NULL,  -- SHA-256 of email
    phone_hash          VARCHAR(64)     NOT NULL,  -- SHA-256 of phone
    device_fingerprint  VARCHAR(255),
    deleted_at          TIMESTAMPTZ     NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_deleted_email ON deleted_accounts (email_hash);
CREATE INDEX idx_deleted_phone ON deleted_accounts (phone_hash);
```

**How it works:**
1. User requests account deletion (Screen D3 → Flow F7)
2. Backend hashes email and phone with SHA-256, inserts into `deleted_accounts`
3. Backend sets `users.status = 'deleted'`, `users.deleted_at = NOW()`
4. Cron job purges rows from `users` where `deleted_at` is older than 30 days
5. On new registration, backend checks if `SHA256(email)` or `SHA256(phone)` exists in `deleted_accounts`
   - If found → reject with message: previously deleted account cannot re-register

Lightweight abuse monitoring. Not a hard block — just logging.

```sql
CREATE TABLE device_registry (
    id                  SERIAL          PRIMARY KEY,
    device_fingerprint  VARCHAR(255)    NOT NULL,
    user_id             VARCHAR(36)     NOT NULL REFERENCES users(user_id),
    device_model        VARCHAR(100),
    os_version          VARCHAR(50),
    registered_at       TIMESTAMPTZ     NOT NULL DEFAULT NOW()
);

CREATE INDEX idx_device_fingerprint ON device_registry (device_fingerprint);
CREATE INDEX idx_device_user ON device_registry (user_id);
```

**Rules from your user flow doc:**
- One device can have multiple user_ids (family sharing a tablet)
- Flag if a single device_fingerprint has > 3 user_ids
- Never hard-block based on device alone

### Edit alembic/env.py (key sections)

```python
import asyncio
from logging.config import fileConfig

from sqlalchemy import pool
from sqlalchemy.ext.asyncio import async_engine_from_config
from alembic import context

# Import your Base and all models
from core.database import Base
from config import get_settings

# Import models so Base.metadata knows about them
from models.user import User  # ★ Week 1
# from models.mission import Mission            # uncomment Week 2
# from models.mission_result import MissionResult  # uncomment Week 2
# from models.deleted_account import DeletedAccount  # uncomment Week 4
# from models.device_registry import DeviceRegistry  # uncomment Week 4

config = context.config
fileConfig(config.config_file_name)

target_metadata = Base.metadata

# Override DB URL from settings
settings = get_settings()
config.set_main_option("sqlalchemy.url", settings.database_url)


def run_migrations_offline():
    url = config.get_main_option("sqlalchemy.url")
    context.configure(url=url, target_metadata=target_metadata)
    with context.begin_transaction():
        context.run_migrations()


def do_run_migrations(connection):
    context.configure(connection=connection, target_metadata=target_metadata)
    with context.begin_transaction():
        context.run_migrations()


async def run_async_migrations():
    connectable = async_engine_from_config(
        config.get_section(config.config_ini_section, {}),
        prefix="sqlalchemy.",
        poolclass=pool.NullPool,
    )
    async with connectable.connect() as connection:
        await connection.run_sync(do_run_migrations)
    await connectable.dispose()


def run_migrations_online():
    asyncio.run(run_async_migrations())


if context.is_offline_mode():
    run_migrations_offline()
else:
    run_migrations_online()
```

### Migration Workflow

```bash
# ── After creating/editing a model file ──

# 1. Generate migration (reads model changes automatically)
docker compose exec backend alembic revision --autogenerate -m "create users table"

# 2. Review the generated file in alembic/versions/
#    Check that upgrade() and downgrade() look correct

# 3. Apply migration
docker compose exec backend alembic upgrade head

# ── Other useful commands ──

# See current migration state
docker compose exec backend alembic current

# See migration history
docker compose exec backend alembic history

# Rollback one step
docker compose exec backend alembic downgrade -1

# Rollback to nothing (empty database)
docker compose exec backend alembic downgrade base
```

## Key Business Logic in the Database Layer

### Session Counting (Flow F5)

Session is consumed **only** when Review phase completes. This happens in `services/user_service.py`:

```python
async def consume_session(db: AsyncSession, user_id: str) -> int:
    """Decrement one free session. Returns remaining count."""
    user = await db.execute(select(User).where(User.user_id == user_id))
    user = user.scalar_one()

    if user.used_free_sessions >= user.max_free_sessions:
        raise NoFreeSessionsError()

    user.used_free_sessions += 1
    user.last_active_at = datetime.now(timezone.utc)
    await db.commit()

    return user.max_free_sessions - user.used_free_sessions
```

The CHECK constraint `chk_used_le_max` in PostgreSQL is the safety net — even if code has a bug, the DB won't allow `used > max`.

### Account Deletion (Flow F7)

```python
import hashlib

async def delete_account(db: AsyncSession, user: User):
    """Soft-delete user and record hashes for abuse prevention."""
    # 1. Record hashes
    deleted = DeletedAccount(
        email_hash=hashlib.sha256(user.email.encode()).hexdigest(),
        phone_hash=hashlib.sha256(user.phone.encode()).hexdigest(),
        device_fingerprint=user.device_fingerprint,
    )
    db.add(deleted)

    # 2. Soft-delete user
    user.status = UserStatus.deleted
    user.deleted_at = datetime.now(timezone.utc)

    await db.commit()
```

### Duplicate Check on Registration

```python
async def check_deleted_account(db: AsyncSession, email: str, phone: str) -> bool:
    """Check if email or phone was previously used by a deleted account."""
    email_hash = hashlib.sha256(email.encode()).hexdigest()
    phone_hash = hashlib.sha256(phone.encode()).hexdigest()

    result = await db.execute(
        select(DeletedAccount).where(
            (DeletedAccount.email_hash == email_hash)
            | (DeletedAccount.phone_hash == phone_hash)
        )
    )
    return result.scalar_one_or_none() is not None
```

## Summary of Rules

1. **One model file per table.** Keep each under 60 lines.
2. **Alembic for every change.** Never run raw `ALTER TABLE` on the database.
3. **All timestamps in UTC** with `TIMESTAMPTZ`. Flutter converts to JST for display.
4. **JSONB for flexible data** (mission content, feedback, conversation logs). Don't over-normalize.
5. **CHECK constraints as safety nets.** Scores 0–100, sessions used ≤ max.
6. **Indexes on foreign keys and common query columns.** Don't over-index early.
7. **Soft-delete users** (status = 'deleted'). Hash personal data for abuse prevention.
8. **Create tables only when you need them.** Week 1 = users only.
